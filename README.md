# Real-Time Traffic Risk Prediction — Kafka · Spark ML · MongoDB

A real-time traffic risk prediction pipeline built on Apache Kafka and Spark Structured Streaming. The system ingests live camera events from roadside checkpoints, detects speed violations across monitored road segments, and layers a Spark ML prediction model on top to identify high-risk vehicles before a new violation is confirmed.
A Random Forest classifier is trained offline on historical violation data and deployed as a Spark PipelineModel for low-latency batch inference on live Kafka streams. Vehicles flagged as HIGH_RISK are added to a 24-hour watchlist — when they reappear at any camera in the network, a proactive alert fires ahead of the rule-based detector.

The design follows the train-offline, infer-in-stream pattern described by Nair et al. (2017), with model interpretability provided via SHAP analysis following Li et al. (2022), and streaming performance tuned using guidance from Maarala et al. (2015) and Kathait et al. (2025).

## Kafka Configuration

Kafka in this project uses a **single-broker, single-partition** setup suitable for local development. The following topics must exist before running any producer or streaming notebook.

### Topic configuration

| Topic | Partitions | Replication factor | Retention | Used by |
|---|---|---|---|---|
| `camera-events-A` | 1 | 1 | Default (7 days) | Producer A → A2 + ML streaming |
| `camera-events-B` | 1 | 1 | Default (7 days) | Producer B → A2 + ML streaming |
| `camera-events-C` | 1 | 1 | Default (7 days) | Producer C → A2 + ML streaming |

### Verify topics exist

```powershell
# Get Kafka container IP
docker inspect kafka_a2 | Select-String "IPAddress"

# List all topics inside the Kafka container
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --list
```

### Create topics if missing

```powershell
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --create --topic camera-events-A --partitions 1 --replication-factor 1
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --create --topic camera-events-B --partitions 1 --replication-factor 1
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --create --topic camera-events-C --partitions 1 --replication-factor 1
```

### Describe a topic (verify config)

```powershell
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --describe --topic camera-events-A
```

### Check consumer group lag (monitor streaming progress)

```powershell
# The ML streaming query uses consumer group 'spark-awas-ml' by default
docker exec kafka_a2 kafka-consumer-groups --bootstrap-server localhost:9092 --describe --group spark-awas-ml
```

### Purge a topic (reset between runs)

```powershell
# Delete and recreate — fastest way to reset offsets
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --delete --topic camera-events-A
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --create --topic camera-events-A --partitions 1 --replication-factor 1
```

### Kafka connectivity troubleshooting

Kafka uses two listener addresses — one for internal Docker networking, one for host access. If producers running on the host cannot connect, it is because `KAFKA_ADVERTISED_LISTENERS` in the container is set to the internal Docker IP rather than `localhost`.

```powershell
# Check what Kafka is advertising
docker exec kafka_a2 env | Select-String "KAFKA_ADVERTISED"
```

The producer notebooks connect using `HOST_IP:9092` where `HOST_IP` is your machine's LAN IPv4 address (not `localhost`). This is required because the producers run in Jupyter inside `spark_a2`, which is a separate container, so `localhost:9092` would attempt to connect to a Kafka broker inside `spark_a2` itself rather than `kafka_a2`.

```powershell
# Get your machine's LAN IPv4 (Windows)
ipconfig | Select-String "IPv4"

# Get your machine's LAN IPv4 (Mac/Linux)
ifconfig | grep "inet " | grep -v 127
```

The ML streaming notebook (`ml_streaming.ipynb`) connects using the Docker bridge network IP of `kafka_a2`, **not** `HOST_IP`. This is because `ml_streaming.ipynb` runs inside `spark_a2`, which can reach `kafka_a2` directly over the Docker bridge.

## Full Setup Instructions

### Step 1 — Start all containers

```powershell
docker start lucid_bohr zealous_shirley kafka_a2 spark_a2
```

Wait approximately 15 seconds for Zookeeper and Kafka to complete their startup handshake before proceeding.

### Step 2 — Verify all containers are running

```powershell
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
```

All four containers must show `Up`. If any show `Exited`, run `docker start <name>` individually and wait 10 seconds.

### Step 3 — Get container IPs

Container IPs are assigned by Docker's bridge network and **change on every Docker Desktop restart**. You must re-run this step at the start of every session.

```powershell
# MongoDB IP (zealous_shirley)
docker inspect zealous_shirley | Select-String "IPAddress"

# Kafka IP (kafka_a2)
docker inspect kafka_a2 | Select-String "IPAddress"
```

Typical values: `172.17.0.2`–`172.17.0.5`. Note both IPs — you will need them in the next step.

### Step 4 — Update IPs in notebooks

Open the following notebooks and update **Cell 3** with the IPs from Step 3:

**`ml_training.ipynb` Cell 3:**
```python
MONGO_URI = "mongodb://<zealous_shirley_IP>:27017/"
```

**`ml_streaming.ipynb` Cell 3:**
```python
MONGO_URI       = "mongodb://<zealous_shirley_IP>:27017/"
KAFKA_BOOTSTRAP = "<kafka_a2_IP>:9092"
```

**`data_design_streaming.ipynb` Cell 3:**
```python
MONGO_URI       = "mongodb://<zealous_shirley_IP>:27017/"
KAFKA_BOOTSTRAP = "<kafka_a2_IP>:9092"
```

**`visualisation.ipynb` Cell 3:**
```python
HOST_IP = "<zealous_shirley_IP>"
```

### Step 5 — Update producer HOST_IP

In **each** producer notebook (`producer_a.ipynb`, `producer_b.ipynb`, `producer_c.ipynb`), update:

```python
HOST_IP = "<your machine LAN IPv4>"   # from ipconfig (Windows) or ifconfig (Mac/Linux)
```

Producers connect to Kafka from outside the Docker network (from `spark_a2`'s Jupyter), so they use the host machine IP rather than the Docker bridge IP.

### Step 6 — Verify Kafka topics exist

```powershell
docker exec kafka_a2 kafka-topics --bootstrap-server localhost:9092 --list
```

If `camera-events-A`, `camera-events-B`, `camera-events-C` are not listed, create them using the commands in the **Kafka Configuration** section above.

### Step 7 — Open Jupyter

Navigate to: **http://localhost:8888**

## Known Limitations and Configuration Assumptions

| Limitation | Detail |
|---|---|
| **Dynamic container IPs** | Docker Bridge IPs change on every Docker Desktop restart. Must re-run `docker inspect` and update `MONGO_URI`/`KAFKA_BOOTSTRAP` in Cell 3 of each notebook at the start of every session |
| **HOST_IP is machine-specific** | Producer notebooks require the local machine's LAN IPv4 address. This differs from the Docker bridge IP used by streaming notebooks |
| **Spark PipelineModel trained on subsampled data** | The Spark MLlib PipelineModel (streaming inference artefact) is trained on a 213k-row subsample. Full-dataset sklearn evaluation metrics (F1=0.9961) reflect the complete training set — both are reported in `ml_training.ipynb` Cell 10 |
| **BBC excluded** | `BalancedBaggingClassifier` could not be benchmarked due to an `imbalanced-learn` API incompatibility in Python 3.8 (`estimator` vs `base_estimator_` attribute conflict). RF + RandomOverSampler already exceeds the Li et al. (2022) BBC-RF benchmark, so this does not affect the comparative conclusion |
| **Binary risk scores (0.0 or 1.0)** | The Spark PipelineModel is trained on a dataset with near-binary segment-level violation rates (99.9% on Segment 1→2). This produces 0/1 probability outputs rather than calibrated values. A production deployment with multi-segment data over longer time windows would yield calibrated intermediate scores |
| **Segment 1→2 has 99.9% violation rate in training data** | This is a dataset composition effect caused by the `camera_event_historic.csv` containing only confirmed violators for Segment 1→2, not data leakage. The `segment_id` feature dominance (MDG=0.511) is legitimate — it reflects the genuinely stricter relative enforcement on the 90 km/h segment |
| **Watchlist 24-hour TTL** | Design decision — risk profiles are reassessed on new evidence per batch rather than accumulated indefinitely. In a production system this would be backed by a MongoDB TTL index |
| **Single-node Docker deployment** | All components run on one physical machine. Production deployment would distribute Kafka, MongoDB, and Spark across separate nodes, enabling the partition-to-core alignment described by Maarala et al. (2015) to provide larger throughput gains |

## Feature Engineering Reference

All features are computed per `(car_plate, date)` aggregate and must be identical across `ml_training.ipynb` and `ml_streaming.ipynb`:

| Feature | Computation | Research justification |
|---|---|---|
| `max_speed_over_limit` | max(speed_reading − speed_limit) per plate-day | Masoud (2024): velocity at measurement point is a top-4 predictor |
| `avg_speed_over_limit` | mean(speed_reading − speed_limit) per plate-day | Masoud (2024): monitoring period average speed |
| `max_speed_ratio` | max(speed_reading / speed_limit) per plate-day | Masoud (2024): normalised speed for cross-segment consistency |
| `segment_id` | 0 = Segment 1→2 (110 km/h), 1 = Segment 2→3 (90 km/h) | Al-Ghamdi et al. (2025) and Li et al. (2022): location is the #1 predictor (SHAP=0.39) |
